### CS1061 — member missing

| Metric | Value |
|---|---|
| Rows seen | 44 |
| Fired (any conf) | 9 (0 match / 9 no-match) |
| Silent | 12 |
| No diag in compile | 23 |
| Production threshold | 0.80 |

All 9 firings are no-match. Spot-check of the 6 emissions at conf ≥ 0.80:

| Conf | Suggestion | Hunk before | Hunk after (agent's actual fix) |
|---|---|---|---|
| 0.90 | `TextBlockElement.TextAlignment` | `.VerticalAlignment(VerticalAlignment.Center)` | `.VAlign(VerticalAlignment.Center)` |
| 0.90 | `TextBlockElement.TextAlignment` | `TextBlock(...).VerticalAlignment(...)` | NumberBox replacement (structural) |
| 0.86 | `TextBlockElement.TextAlignment` | `.Title()` on TextBlock | `Heading(...)` (structural) |
| 0.86 | `TextBlockElement.TextAlignment` | `.Title()` on TextBlock | `Heading(...)` (structural) |
| 0.82 | `TextBlockElement.Set` | `.Style(Theme.TitleTextBlockStyle)` | `.FontSize(28).SemiBold()` (structural) |
| 0.82 | `TextBlockElement.Set` | DataGrid replacement | (full rewrite) |

**Pattern:** the agent writes WinUI-style API names (`.VerticalAlignment`, `.Style`, `.HorizontalAlignment`) that don't exist on Reactor `*Element` types. The correct member is a Reactor shortcut (`VAlign`, `HAlign`, fluent helpers like `.SemiBold()`, `.FontSize()`) whose name is too far from the WinUI name for JaroWinkler to find. Our fuzzy match instead picks the closest-by-edit-distance sibling (`TextAlignment`, `Set`), which is a real member but the wrong one.

**Diagnosis:** fundamentally a Tier-3 rule problem. Tier-2 fuzzy match can't bridge "VerticalAlignment → VAlign" because JaroWinkler("VerticalAlignment", "VAlign") ≈ 0.55, well below the 0.70 similarity floor. We fall through to the second-closest match and emit it. The right fix is rule authoring under Phase 3.

**Decision:** hold at 0.80. The diagnostic-count gate provides the main protection (small typo-only builds are filtered out, where wrong CS1061 suggestions would do the most damage). Raising the threshold higher would:
- Silence the canonical `Button("x").OnClick(...)` factory-param case (currently emits at conf 0.90).
- Force the `Brr → Bar` synthetic test (currently asserts conf ≥ 0.85) to be rewritten against `SuggestRaw`.
- Still not give us right answers on the empirical corpus — the right answers don't live in the fuzzy-match neighborhood.

This is the strongest single Phase-3 motivation in the corpus.

### CS0117 — no static member on type

| Metric | Value |
|---|---|
| Rows seen | 44 |
| Fired (any conf) | 13 (0 match / 13 no-match) |
| Silent | 21 |
| No diag in compile | 10 |

Spot-check of the 5 firings at conf = 1.00:

| Conf | Suggestion | Hunk before | Hunk after (agent's actual fix) |
|---|---|---|---|
| 1.00 | `Theme.Background` | `Theme.AppBackground` | `Theme.SolidBackground` |
| 1.00 | `Theme.Background` | `Theme.AppBackground` | (structural) |
| 1.00 | `Theme.Background` | `Theme.AppBackground` | `Theme.SolidBackground` |
| 1.00 | `Theme.Background` | `Theme.AppBackground` | `Theme.SolidBackground` |
| 1.00 | `Theme.Background` | `Theme.AppBackground` | `Theme.SolidBackground` |

(Plus four conf=0.88 emissions of `Theme.get_Background` — property-accessor synthesized methods leaking through. **Fixed in this branch:** `CollectStaticMembers` and `CollectInstanceMembers` now skip non-ordinary `IMethodSymbol`s — see `src/Reactor.Cli/Check/Suggesters/SymbolSuggester.cs:283-299`.)

**Pattern:** agent writes `Theme.AppBackground` / `Theme.DefaultBackground` (non-existent). Suggester proposes `Theme.Background` (shortest-edit-distance real member). Correct answer is `Theme.SolidBackground` (sibling member with completely different stem). Conf=1.00 comes from the formula: high base similarity (`AppBackground` ↔ `Background` JaroWinkler ≈ 0.85) + strong margin + Reactor-type bonus → caps at 1.00. So conf=1.00 here does **not** mean "definitely right"; it means "definitely a real member, in a Reactor type, with no close competitor." For systematic API confusion like this, that's not enough signal.

**Cluster reference:** `patterns.json` C0019 (CS0117 / Theme / other, count=16, frequency=1.6%) is exactly this pattern. Top Phase-3 rule candidate.

**Decision:** hold at 0.75 with the gate as the safety net. Phase-3 rule "On CS0117 against Theme with missing-member-ending-in-`Background`, suggest `Theme.SolidBackground`" replaces the fuzzy-match path. (Property-accessor leakage — the smaller suggester nit — is already fixed in this branch.)

## Top Phase-3 rule targets

In order of cluster frequency where Tier-2 produces wrong-direction suggestions:

1. **CS0117 / Theme — Background suffix lookup.** Cluster C0019, 16 events, 1.6%. The agent's most common Theme mistake is reaching for an English-plausible name (`AppBackground`, `DefaultBackground`) and getting back a worse fuzzy match. Rule: small lookup table from common-wrong-name → canonical Reactor token.
2. **CS1061 / *Element — WinUI → Reactor name family.** Cluster C0017 + others, ~22 events combined. Rule family: `.VerticalAlignment(x)` → `.VAlign(x)`, `.HorizontalAlignment(x)` → `.HAlign(x)`, `.Style(...)` → fluent shortcut family, etc. May want a single rule with a small mapping table rather than one rule per pair.
3. **CS1955 / GridSize — missing parens.** Cluster C0004, 110 events, 10.7%. Largest single bucket in the corpus. Rule: detect method-group-as-value where the method is a Reactor factory (`GridSize.Star`, etc.) and suggest the parenthesized form.

The cross-agent reproducibility bar (Validation Gate #2) is currently unmet — every rule above will need a second-agent corpus drop or a documented justification before merging. Recommend kicking off a second corpus run on a different model (`claude-sonnet-4-6`?) before opening Phase-3 rule PRs.
